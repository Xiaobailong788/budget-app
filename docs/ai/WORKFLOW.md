# 🔄 通用作业工作流

> 本文档描述在 **任何类型任务** 下，AI Agent 生态的协作方式。
> 不限于编程。适用于：开发功能、写文档、做研究、整理文件、设计方案 — 任何你需要 AI 完成的事。
>
> **核心思想：** Director 编排一切。Director 只规划、分派、审查，绝不执行。
> 执行层由 Team-Leader 统管，Team-Leader 根据任务类型选择合适的执行 Agent。

---

## 🧩 Agent 生态全景

```
                        ┌──────────────┐
                        │    User      │  ← 需求提出者 + 终审决策者
                        └──────┬───────┘
                               │
                        ┌──────▼───────┐
                        │   Director   │  ← 编排者：规划·分派·审查（绝不执行）
                        └──────┬───────┘
                               │ task
                        ┌──────▼───────┐
                        │  Team-Leader │  ← 执行协调员：拆解·分派·汇总·写日志
                        └──────┬───────┘
                 ┌──────────────┼──────────────┐
                 │              │              │
          ┌──────▼─────┐ ┌─────▼─────┐ ┌──────▼─────┐
          │  Researcher │ │   Writer  │ │   Tester   │  ← 专业执行 Agent
          │ (搜索/调研)  │ │ (写/改文档)│ │ (验证/测试) │
          └────────────┘ └───────────┘ └────────────┘
          ┌──────▼─────┐ ┌─────▼─────┐ ┌──────▼─────┐
          │  UI-Agent  │ │ Organizer │ │  Explore   │  ← 更多专业 Agent
          │ (格式美化)  │ │ (文件整理) │ │ (快速搜索)  │
          └────────────┘ └───────────┘ └────────────┘
```

### 角色速览

| 角色 | 可用的工具 | 一句话职责 |
|------|-----------|-----------|
| **User** | — | 提需求、做决策、终审验收 |
| **Director** | `task`, `question` | 规划、分派、审查 — 绝不执行 |
| **Team-Leader** | `task`, `read`, `write`, `edit`, `glob`, `grep`, `question` | 执行协调 — 拆解任务、分派或自干、汇总、写日志 |
| **Explore** | `read`, `glob`, `grep`, `bash`(有限) | 快速探索代码库 |
| **Researcher** | `webfetch`, `websearch` | 只搜索网页，绝不干别的 |
| **Writer** | `read`, `write`, `edit`, `glob`, `grep` | 写和改文档，不跑代码 |
| **Tester** | `read`, `bash`, `glob`, `grep` | 只测试和报告，绝不修复 |
| **UI-Agent** | `read`, `write`, `edit` | 美化格式，不改变内容 |
| **Organizer** | 读写/移动文件 | 组织文件结构 |

> **注意：** 每个 Agent 都有严格**工具闸门**，不能越权。
> 例如 Researcher 不能写文件，Writer 不能运行代码，Director 不能碰任何执行工具。

## ⚙️ 核心工作流（通用，不限场景）

这个流程适用于**任何类型的任务** — 不管是写代码、写文章、做调研、还是整理文件。

### 阶段 ①：接收与澄清

**参与者：** User → Director
**产出：** 清晰的需求描述

1. User 提出需求（可能是任何事：开发功能、写文档、做研究、改配置……）
2. Director **不做任何执行操作**（不读文件、不搜索、不写代码）
3. Director 评估需求清晰度：
   - ✅ 清晰 → 直接进入规划
   - ❌ 不清晰 → 用 `question` 提问澄清
     - 一次一个问题
     - 优先用多选
     - 聚焦：目标是什么？边界在哪？成功标志是什么？不想要什么？
4. 如果需求包含多个独立子系统 → 建议拆分为子项目，逐个处理

### 阶段 ②：规划

**参与者：** Director
**产出：** 任务列表（带 agent-type 标签、依赖、验收标准）

1. Director 将需求拆解为**可独立执行的原子任务**
2. 每个任务包含：
   - **描述**：具体做什么
   - **agent-type 标签**：指明谁最适合做（`explore` / `researcher` / `writer` / `tester` / `general` / `organizer`）
   - **验收标准**：怎样算完成
   - **依赖**：哪些任务必须先完成
3. 确定任务之间的依赖关系，规划并行/串行策略
4. **不需要写设计文档的场景**（如简单的 bug fix、小改动）→ 跳过设计阶段，直接出任务列表

### 阶段 ③：设计（可选，适用于复杂任务）

**参与者：** Director → User
**产出：** 批准的设计方案

*仅当任务复杂度需要设计方案时才走此阶段。*

1. Director 探索项目上下文（通过 Team-Leader 调 Explore Agent 做代码搜索）
2. 提出 2-3 种方案及推荐
3. User 确认方案
4. Director 分节呈现设计，逐节获得 User 批准

### 阶段 ④：交办

**参与者：** Director → Team-Leader
**产出：** Team-Leader 收到的完整任务包

1. Director 通过 `task` 调用 Team-Leader
2. 传递内容：
   - 完整任务列表（含 agent-type、依赖、验收标准）
   - 上下文信息（用户需求、方案选择、设计文件路径）
   - 本项目的 AI 编程规范位置：`docs/ai/`
3. Director **STOP 等待结果** — 不做任何执行工作

### 阶段 ⑤：执行（由 Team-Leader 驱动）

**参与者：** Team-Leader → 各 Sub-Agent
**产出：** 完成的交付物 + 日志

Team-Leader 的 5 步强制流程（详见 Team-Leader 自己的 prompt）：

1. **理解** — 读 Director 的任务列表，不明确则退回提问
2. **拆解** — 每个任务拆成原子工作单元，写 Assignment Plan
3. **执行** — 按依赖顺序分派或自干
   - 无依赖的单元并行执行
   - 根据 agent-type 标签选择合适的 Sub-Agent
   - 每个单元完成后收结果 + 检查
4. **最终检查** — 如有代码调 Tester 验证，交付物调 UI-Agent 美化
5. **日志 + 交付** — 写技术日志 + 用户日志，打包返回 Director

### 阶段 ⑥：审查

**参与者：** Director
**产出：** 审查结论（合格/需返工）

收到 Team-Leader 结果后，逐一检查：

□ 是否满足 User 的所有需求？
□ 是否有遗漏的任务？
□ 执行质量如何？（如果是代码：安全/性能/风格）
□ 日志是否完整？（技术日志 + 用户日志）
□ 各 Agent 是否有越权操作？
□ 整体交付物是否前后一致？

- ❌ 有任何一项不合格 → 退回 Team-Leader 返工（附上具体要求）
- ✅ 全部合格 → 交付给 User

### 阶段 ⑦：交付

**参与者：** Director → User
**产出：** User 确认的最终状态

1. Director 向 User 呈现交付物摘要
2. 说明做了什么、为什么、注意事项
3. User 验收或提出修改意见

## 🌊 流程弹性说明

> 以上 7 个阶段是**完整版流程**。实际执行时可以根据任务复杂度裁剪：

| 任务类型 | 走哪些阶段 |
|---------|-----------|
| **修一个 bug / 改一行文案** | ① → ②(简略) → ④ → ⑤ → ⑥ → ⑦ |
| **写一篇文档 / 做一项研究** | ① → ② → ④ → ⑤ → ⑥ → ⑦ |
| **新增一个小功能** | ① → ② → ③(简略) → ④ → ⑤ → ⑥ → ⑦ |
| **大版本 / 大功能 / 重构** | ① → ② → ③(完整) → ④ → ⑤ → ⑥ → ⑦ |
| **整理文件目录** | ① → ②(简略) → ④ → ⑤ → ⑥ → ⑦ |

**关键原则：**
- Director **永远不做执行** — 无论任务多小
- 需求不清时 **必须先问清楚**，不要猜
- 任务列表必须包含 **agent-type 标签**，让 Team-Leader 知道派谁
- 审查是 Director 的**核心职责**，不能跳过

## 🚦 决策权分布

| 谁决策 | 决策什么 |
|--------|---------|
| **User** | 需求是否值得做、优先级、选方案 A/B/C、设计是否批准、最终验收 |
| **Director** | 任务如何拆解、分派给谁、执行结果是否合格 |
| **Team-Leader** | 原子单元如何进一步拆解、哪个 Sub-Agent 执行、执行顺序 |
| **AI 集体** | B/C 级版本号递增（A 级必须 User 决定） |

## 🛠️ Agent-Type 标签使用指南

Director 在规划阶段给每个任务打 `agent-type` 标签，指导 Team-Leader 选择合适的执行 Agent。

| 标签 | 适合的任务类型 | 举例 |
|------|--------------|------|
| `explore` | 快速搜索代码、查找文件、了解项目结构 | "找到所有使用了 X 的地方" |
| `researcher` | 搜索网页、获取外部信息 | "查找最新的 CSS 语法规范" |
| `writer` | 写文档、改文档、润色 | "写设计文档"、"更新 README" |
| `tester` | 运行测试、验证正确性 | "跑测试看看有没有报错" |
| `general` | 通用的多步骤任务 | 常规的编码、配置、重构 |
| `organizer` | 组织文件、移动/重命名 | "把文件按新结构重新组织" |
| `ui-agent` | 格式美化 | "统一 markdown 格式" |
| `team-leader` | 需要再协调多个 Sub-Agent | （Director 只用 task 调 Team-Leader） |
