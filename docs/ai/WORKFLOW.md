# 🔄 广义开发工作流

> 本文档描述本项目的完整开发周期：谁参与、做什么、什么顺序、谁决策。
> 适用于新增功能、大版本迭代、重要重构等场景。

---

## 👤 角色定义

| 角色 | 是谁 | 职责 |
|------|------|------|
| **用户 (User)** | 项目所有者 | 提出需求、验收功能、做终审决策 |
| **Director** | AI Agent（当前） | 理解需求、拆解任务、分派工作、审查结果 |
| **Team-Leader** | AI Agent（执行层） | 接收任务、协调执行、汇总结果、写日志 |
| **Researcher** | AI Agent（研究型） | 只有只读能力，用于探索代码、搜索模式 |
| **Writer** | AI Agent（写作型） | 编写文档、撰写设计文档、用户日志 |
| **Tester** | AI Agent（验证型） | 运行验证、检查正确性、报告问题 |
| **UI-Agent** | AI Agent（UI 美化） | 美化 UI 呈现，不改变内容 |
| **Organizer** | AI Agent（整理型） | 组织文件结构、移动/重命名 |

## 📐 完整开发阶段

### 阶段 0：需求理解
**参与角色：** User → Director
**产出：** 清晰的需求描述

1. User 提出需求（功能描述、问题描述、变更请求）
2. Director ：
   - 如果需求不清 → 提问澄清（一次一个问题，优先多选）
   - 如果需求超出范围 → 标记为独立项目建议拆分
   - 如果需求可执行 → 进入下一阶段

### 阶段 1：探索与设计
**参与角色：** Director → Researcher → Director
**产出：** 设计文档（`docs/superpowers/specs/YYYY-MM-DD-xxx-design.md`）

1. Director 先探索项目上下文（文件结构、文档、最近提交）
2. Researcher 做定向代码搜索（如果需要）
3. Director 提出 2-3 种方案及推荐
4. User 确认方案
5. Director 分节呈现设计，逐节获得 User 批准
6. User 批准完整设计

### 阶段 2：设计文档化
**参与角色：** Director → Writer
**产出：** 正式的设计文档，已 git commit

1. Director 将批准的设计写成 spec 文档
2. Director 做 spec self-review（占位符检查、一致性、范围、歧义）
3. User 审阅文档
4. 如有修改 → 修改后重新审阅，直到 User 批准

### 阶段 3：实施计划
**参与角色：** Director → （写作型 Agent / 自己执行）
**产出：** 实施计划

1. Director 将设计拆解为可执行的任务列表
2. 每个任务包含：描述、依赖、验收标准、涉及的文件
3. 确定任务的先后顺序和并行可能性

### 阶段 4：执行与构建
**参与角色：** Team-Leader → 各执行 Agent
**产出：** 可工作的代码变更

1. Team-Leader 按依赖顺序分派任务
2. 独立任务可并行执行
3. 每条规则（RULES.md #1-#18）必须在执行过程中遵守
4. 每个任务完成后执行 `bash build.sh` 验证构建成功
5. 检查 `index.html` 在浏览器中可正常打开

### 阶段 5：验证与审查
**参与角色：** Team-Leader → Tester → Director
**产出：** 审查通过的代码

1. Tester 运行验证（功能测试、边界情况、回归检查）
2. Director 审查：
   - □ 是否满足所有需求
   - □ 是否有遗漏任务
   - □ 代码质量（安全/性能）
   - □ 日志是否完整（技术日志 + 用户日志）
   - □ 是否有越权操作
   - □ 前后一致
3. 不合格 → 退回 Team-Leader 返工
4. 合格 → 交付给 User

### 阶段 6：文档收尾
**参与角色：** Team-Leader → Writer
**产出：** 完整的文档更新

1. 更新 `STRUCTURE.md`（函数地图、新文件、新章节）
2. 写 `technical/YYYY-MM-DD-xxx-technical.md`（技术日志）
3. 写 `user/YYYY-MM-DD-xxx-user.md`（用户变更日志）
4. 更新 `README.md`（功能列表、版本号、变更说明）

### 阶段 7：交付
**参与角色：** Director → User
**产出：** 用户确认的完成状态

1. git commit 所有变更
2. Director 向 User 交付结果摘要
3. User 验收或提出修改意见

## 🚦 决策节点

| 节点 | 决策者 | 决策内容 |
|------|--------|----------|
| 需求进入 | User | 这个需求值得做吗？优先级如何？ |
| 方案选择 | User | 选方案 A/B/C？ |
| 设计批准 | User | 设计是否完整正确？ |
| 实现策略 | Director/Team-Leader | 如何拆解任务？哪些任务可并行？ |
| 代码审查 | Director | 代码是否合格？哪些需要返工？ |
| 版本号 | AI（B/C）/ User（A） | 这次变更属于哪个级别？ |
| 最终验收 | User | 功能是否满足需求？是否可以发布？ |

## 📊 角色调用矩阵

| 需要做的事 | 调用谁 |
|-----------|--------|
| 理解需求、拆任务、分派、审查 | Director |
| 执行任务、收集结果、写日志 | Team-Leader |
| 搜索代码、查找模式（只读） | Researcher |
| 运行验证、检查正确性 | Tester |
| 写设计文档、技术文档 | Writer |
| 美化 UI 呈现 | UI-Agent |
| 组织文件结构、重命名/移动 | Organizer |
